MOOD-TO-MEAL BUTLER: An AI-Powered Emotional Intelligence Meal Recommendation System

PROJECT OVERVIEW

Mood-to-Meal Butler is an intelligent conversational AI agent that detects user emotions and recommends personalized meal suggestions based on emotional state, preferences, weather conditions, and dietary requirements. Built on Google ADK 2.0 Workflow, the system uses multi-modal emotion detection, mood-aware meal filtering, and continuous conversation loops to provide emotionally intelligent food recommendations.

The core mission: Help people discover meals that nourish both body and mind by understanding not just what they like to eat, but how they feel, and what they need nutritionally and emotionally.

PROBLEM STATEMENT

Most meal recommendation systems work transactionally—users search for recipes or restaurants based on cuisine type or rating. But food is deeply emotional. When someone is tired, they need comfort and energy. When stressed, they need calming, grounding meals. When celebrating, they need exciting, shareable dishes.

Current solutions lack emotional context. They don't ask "How are you feeling?" They don't understand that a sad person needs chocolate or comfort food, not just any dessert. They don't adapt recommendations based on weather, time constraints, or energy levels.

Mood-to-Meal Butler solves this by treating food as emotional medicine—combining AI emotion detection, mood-aware meal curation, and continuous conversation to recommend meals that truly take care of the user.

KEY FEATURES

1. NATURAL LANGUAGE EMOTION DETECTION
Users don't need commands. They just type: "i am tired" or "i'm feeling stressed"
The system automatically detects 26 emotion categories (joy, sadness, exhaustion, stress, anger, fear, surprise, love, etc.) using keyword matching with 150+ emotion keywords across English and Vietnamese.
Detects not just emotions but situations: "working late", "celebrating", "sick", "at gym", "with family"

2. THREE INTERACTION PATHS
Path A: Direct Emotion Input ("i am tired") → Instant 9 meal suggestions
Path B: Full Interview (/dailyfood) → 6 guided questions → 3 personalized meals
Path C: Direct Command (/goal happy) → Mood specification → 9 meals

Users choose their comfort level. New users might prefer the interview. Returning users want instant detection.

3. MOOD-AWARE MEAL FILTERING
The system maintains a curated database of 274 dishes with rich metadata:
- Nutritional rules (energy level, protein content, vitamins)
- Mood tags (which emotions benefit from which foods)
- Regional origins (Vietnamese, Asian, Global)
- Health tags (vegan, keto, gluten-free, high-protein)
- Time requirements (quick, normal, leisurely)
- Budget categories (cheap, moderate, expensive)

When a user says "i am tired", the system doesn't just find "comfort food"—it specifically retrieves meals high in protein, iron, and B vitamins, matched to the "exhaustion" emotion.

4. WEATHER-AWARE RECOMMENDATIONS
The system fetches real-time weather data and adapts recommendations. Cold weather → warm, nourishing meals. Hot weather → refreshing, hydrating options. Rainy day → comfort food. This contextual awareness makes recommendations feel personally tailored.

5. CONTINUOUS CONVERSATION LOOP
Users aren't limited to one recommendation per session. After enjoying a suggested meal, they can immediately ask for another: "Now I'm happy, what should I eat?"
The system loops back to the greeting, maintaining chat history and meal history to avoid repetitive suggestions.

6. PERSISTENT MEAL HISTORY & DIARY
Every meal recommendation is saved to a SQLite database with timestamp, mood, weather conditions, and user feedback.
Users can review their "food diary"—seeing patterns like "I always want comfort food when tired" or "I eat lighter meals on hot days".
This historical data trains personalization over time.

7. MULTI-LANGUAGE SUPPORT
English and Vietnamese support built-in. Interview questions, meal descriptions, emotional responses—all localized.
Extensible for additional languages.

TECHNICAL ARCHITECTURE

Built on Google ADK 2.0 Workflow (Agents Development Kit), the system uses an event-driven DAG (Directed Acyclic Graph) with 10 interconnected nodes:

1. init_db: Initialize SQLite database for meal history
2. load_history_via_mcp: Fetch user history using MCP (Model Context Protocol)
3. fetch_weather: Get real-time weather via OpenWeather API
4. butler_interview: HITL (Human-in-the-Loop) conversation - captures user input via RequestInput
5. security_check: Validate input for prompt injection attacks
6. llm_suggest: Core recommendation engine - uses mood_service or LLM
7. human_pick: Let user choose from 9 (or 3) suggestions
8. generate_output: Format meal details with ingredients, restaurants, nutrition
9. write_diary_entry: Log to daily diary file
10. record_session: Save to SQLite history

KEY WORKFLOW:

User Input → butler_interview (captures emotion/mood via RequestInput)
         ↓
     security_check (validate)
         ↓
     llm_suggest (detect emotion → mood_service → get 9 meals)
         ↓
     human_pick (user selects 1-9)
         ↓
     generate_output (show details)
         ↓
     write_diary_entry & record_session (save)
         ↓
     [LOOP] butler_interview (continuous conversation)

CRITICAL IMPLEMENTATION CHALLENGES & SOLUTIONS

Challenge 1: Input Capture in RequestInput Workflows
Initial problem: butler_interview wasn't waiting for user input. It showed a greeting and immediately passed empty data downstream, causing all recommendations to fail.

Solution: Implemented rerun_on_resume pattern. First call shows greeting + RequestInput. On resume (after user input), system retrieves message from ctx.resume_inputs and routes to appropriate node.

Challenge 2: State Persistence Across RequestInput Interrupts
Problem: When butler_interview yields RequestInput, the workflow pauses. When it resumes, context can be lost.

Solution: Store critical data in ctx.state before any yield(). Data in ctx.state survives RequestInput interrupts. Example: mood_service recommendations stored in ctx.state["suggestions"] before yielding.

Challenge 3: Continuous Conversation Loop
Problem: After one meal recommendation, the workflow ended. Users couldn't ask for another meal.

Solution: Added workflow edge: record_session → butler_interview. After saving meal to database, loop back to conversation start. This enables unlimited meal requests in one session.

Challenge 4: Null/Empty Recommendations
Problem: mood_service returned 9 meals but they weren't passed to human_pick, showing error "I couldn't generate meal suggestions".

Solution: Store recommendations in both output dict AND ctx.state:
  suggestions = recs (from mood_service)
  ctx.state["suggestions"] = suggestions
  yield Event(output={"suggestions": suggestions}, state={"suggestions": suggestions})

Challenge 5: Natural Language vs Command Parsing
Initial design required /goal prefix. User feedback: "I don't want to type /goal tired, I just want to say I'm tired"

Solution: Added detect_emotion() function that auto-detects emotion keywords in any user message. No special syntax needed. "i am tired" is automatically detected as exhaustion emotion.

Challenge 6: Error Handling in State Cleanup
Problem: Code used `del ctx.state["key"]` which throws AttributeError if key doesn't exist.

Solution: Changed to `ctx.state.pop("key", None)` which safely deletes or returns None if key missing.
